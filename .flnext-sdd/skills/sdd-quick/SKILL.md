# SDD 快速通道

小需求/配置调整的加速通道，合并部分阶段但保留核心门控。

> ⚠️ **P0 生产事故、零日漏洞、数据恢复** → 必须使用 `/flnext-sdd-hotfix`，不得使用 Quick 通道。

## 适用场景

| 场景 | 示例 | 应使用的通道 |
|------|------|-------------|
| 小需求 | 单个页面/接口变更 | ✅ Quick |
| 配置调整 | 修改配置项/文案 | ✅ Quick |
| 依赖升级 | 小版本/安全补丁 | ✅ Quick |
| P0 生产事故 | 服务宕机、数据损坏 | ❌ 用 `/flnext-sdd-hotfix` |
| 零日漏洞 | CVSS ≥ 9.0 被利用 | ❌ 用 `/flnext-sdd-hotfix` |

## 快速通道流程

```
标准流程:     Discovery → … → 测试 → 7b 集成门控 → 提测 → 验收 → 发布
快速通道:     [Quick Scope] → [Quick Design] → [Quick Code] → [Quick Test] → 7b → 提测 → 验收 → 发布
```

## 执行步骤

### Step 1: 快速范围确认

1. 询问: "这个修改的范围是什么？"
2. 确认影响范围:
   - 仅后端
   - 仅前端
   - 前后端都需要
3. 评估是否适合快速通道:
   - 修改文件 < 10 → 适合
   - 修改文件 10-20 → 需确认
   - 修改文件 > 20 → 不适合，建议走标准流程

### Step 2: 快速设计

1. 用一段文字描述变更方案
2. 列出受影响的文件和 API
3. 用户确认方案

### Step 3: 快速编码

1. 直接实现代码变更
2. 仍需通过双编译门禁（Phase A/B）
3. 原子提交

### Step 4: 快速测试

1. 编写关键路径测试用例
2. 执行测试
3. 确认通过

### Step 5: 标准集成门控 / 提测 / 验收 / 发布

1. 运行 `/flnext-sdd-integration-gate`（双编译 + 联调，不可跳过）
2. 后续与标准流程一致（8-submit → 9-accept → 10-release）

## 限制

1. **不跳过编译门禁** — Phase A/B 门控仍然强制
2. **不跳过提测** — 必须走标准提测流程
3. **不跳过验收** — 必须获得产品确认
4. **文档简化** — 产出简化版文档而非完整文档
