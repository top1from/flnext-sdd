# 会诊模式 (Consultation Mode)

## 概述

会诊模式是 FLNext-SDD v4.0 的受控多 Agent 协作机制。当单一 Agent 的视角不足以解决复杂问题或歧义过高时，系统自动召集相关领域 Agent 进行**结构化分析**，产出**明确结论**。

> 会诊模式**不是**头脑风暴，**不是**自由讨论，**不是**民主投票。它是**受控的专业会诊**，输出必须是结构化的、可执行的结论。

## 触发条件

1. **歧义评分 > 0.5**（config.yaml `consultation_mode.ambiguity_threshold`）
2. **决策冲突**：两个同样合理的方案，需要额外视角
3. **风险评估**：检测到潜在高风险，需要多维度评估

**不会触发**：阶段门控确认、已明确的问题、紧急修复（走 `/flnext-sdd-hotfix`）

## 会诊流程

### Step 1: 问题结构化

```markdown
# 会诊请求

**触发阶段**: Phase {N} — {Phase名称}
**歧义评分**: {0.0-1.0}
**核心问题**: {一句话描述}
**已知约束**: {必须遵守的约束条件}
**期望输出**: {需要会诊提供的具体信息}
```

### Step 2: 专家选择

| 问题类型 | 必选专家 | 可选专家 |
|---------|---------|---------|
| 架构设计 | Atlas | Shadow, Iron |
| 需求歧义 | Luna (PM) | Max (UX), Hawk |
| 技术选型 | Atlas | Iron, Pixel |
| 性能瓶颈 | Iron | Atlas, Hawk |
| 安全漏洞 | Shadow | Iron, Atlas |

### Step 3: 独立分析

每个专家独立输出：核心判断、关键风险、推荐方案、前提假设（各不超过 300 字，必须有明确推荐）。

### Step 4: 结论合成

负责 Agent 综合输出《会诊诊断报告》，含共识点、分歧点、最终建议、决策依据链。

### Step 5: 人工确认

人类选择：**接受建议** / **要求补充分析** / **否决并给出方向**。会诊本身不产生门控确认。

## 约束规则

1. 会诊不是门控的替代品
2. 单个会诊不超过 2 轮专家分析
3. 必须有结论，无结论视为失败
4. 报告归档至 `docs/sdd/consultations/`
