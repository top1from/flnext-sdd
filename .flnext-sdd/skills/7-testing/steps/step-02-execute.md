# Step 2: 编译复检 + 执行测试用例

## ⚠️ 测试前编译复检（强制）

### 在执行任何测试用例之前，必须重新编译

```
测试阶段的代码可能已经被修改过（修复 Bug、调整测试环境）。
编译复检确保测试面对的是可工作的代码。
```

### 2.0 强制编译复检

```bash
# 后端编译
cd {backend_dir}
npm run build          # 或 mvn compile / go build
# 确保退出码为 0

# 前端编译
cd {frontend_dir}
npm run build          # 或 yarn build / pnpm build
# 确保退出码为 0
```

**IF 编译失败:**
```
🚨 编译复检失败 — 代码在测试阶段被改坏了

编译输出: {完整错误}

说明: 测试阶段可能引入了编译错误。需要先修复编译问题才能继续测试。

❌ 禁止操作:
  - 跳过编译直接测试
  - 声称"联调通过"但编译未通过
  - 降低编译标准

→ 请确认后修复编译错误
→ 或输入 "S" 保存进度稍后处理
```

**编译通过后才能继续执行测试用例。**

---

## 2.1 执行测试用例

按照测试用例文档逐个执行测试，记录结果。

### ⚠️ 实现代码只读规则

```
测试阶段的角色: 验证者，不是修改者

✅ 允许操作:
  - 执行测试用例
  - 记录测试结果
  - 分析失败原因
  - 上报问题 (ESCALATE)

❌ 禁止操作:
  - 修改业务代码（Controller/Service/Repository/Component）
  - 修改 API 实现
  - 修改数据库结构
  - 任何可能影响编译结果的代码变更

测试发现的实现 Bug → ESCALATE → 人工决定如何修复
```

## 2.2 执行流程

对每个测试用例:
```
执行 TC-001:
- 前置条件检查: [满足/不满足]
- 步骤执行:
  1. [操作] → [实际结果] → [PASS/FAIL]
  2. [操作] → [实际结果] → [PASS/FAIL]
- 测试状态: [PASS/FAIL/BLOCK]
- 备注: [如有问题]
```

## 2.3 问题处理

**测试失败 → 实现 Bug？**

```
🔴 测试失败: TC-003

失败步骤: {描述}
预期结果: {预期}
实际结果: {实际}

→ 这是实现层面的 Bug，非测试环境问题
→ ESCALATE 给开发人员
→ 不会自动修改业务代码来修复此 Bug
→ 继续执行剩余测试用例
```

**测试失败 → 测试本身有问题？**
```
🟡 测试问题: TC-005

问题: 测试步骤描述不清晰 / 预期结果不准确

→ 标记状态为 BLOCK
→ 不会修改实现来迎合测试
→ 需要人工确认测试用例是否需要修改
```
