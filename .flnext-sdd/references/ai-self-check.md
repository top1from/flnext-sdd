# AI 生成代码强制自检清单

> 提测前必须执行。AI 需逐条检查本分支所有变更代码，将结果附在 MR 描述中。
> 每项检查结果为 ✅ 通过 / ⚠️ 需关注 / ❌ 不通过，不通过必须有说明。

---

## 一、性能与复杂度

| # | 检查项 | 判定标准 |
|---|--------|---------|
| 1.1 | 是否有循环或递归？最坏情况迭代次数？ | 嵌套循环 > 2 层 → ⚠️；递归无终止条件 → ❌ |
| 1.2 | 是否有排序、查找等算法？时间复杂度？ | O(n²) 及以上的数据量敏感操作 → ⚠️ |
| 1.3 | 是否有大量字符串拼接或正则？ | 循环内字符串 += → ⚠️；正则未预编译 → ⚠️ |
| 1.4 | 输入数据量变为 100 倍，代码会明显变慢吗？ | 需根据实际复杂度判断，O(n) 及以上可能受影响 |
| 1.5 | 是否有不必要的对象创建（内存分配热点）？ | 循环内 new 大对象 → ⚠️ |
| 1.6 | 是否有大文件上传/下载？是否支持分块/流式？ | 全量加载大文件进内存 → ❌ |

## 二、数据库与存储

| # | 检查项 | 判定标准 |
|---|--------|---------|
| 2.1 | 是否有数据库查询？是否使用了索引？ | 无索引的 WHERE/JOIN 字段 → ⚠️ |
| 2.2 | 是否存在 N+1 查询问题？（循环内逐条查询） | 循环内执行 SQL → ❌ |
| 2.3 | 是否有慢查询风险？是否缺少 LIMIT？ | 全表扫描无分页 → ❌ |
| 2.4 | 事务边界是否合理？是否有长事务？ | 事务内包含外部 API 调用 → ❌ |
| 2.5 | 数据库连接是否正常释放？是否使用连接池？ | 手动管理连接未释放 → ⚠️ |
| 2.6 | 是否有 SQL 注入风险？（拼接 SQL 字符串） | 直接拼接用户输入到 SQL → ❌ |

## 三、外部 API 与网络

| # | 检查项 | 判定标准 |
|---|--------|---------|
| 3.1 | 是否有外部 API 调用？是否有超时设置？ | 无 timeout → ❌ |
| 3.2 | 是否有重试机制？退避策略是否合理？ | 无重试 + 关键业务 → ⚠️；无限重试 → ❌ |
| 3.3 | API 调用失败时是否有降级/兜底方案？ | 关键路径无 fallback → ⚠️ |
| 3.4 | 是否有请求合并/批量化机会？ | 循环内逐条调 API → ⚠️ |
| 3.5 | 是否处理了 HTTP 错误状态码？ | 只处理 200，其他吞掉 → ❌ |

## 四、并发与线程安全

| # | 检查项 | 判定标准 |
|---|--------|---------|
| 4.1 | 是否有共享状态或全局变量？ | 多线程下无锁保护 → ❌ |
| 4.2 | 是否存在竞态条件（race condition）？ | 先查后改（check-then-act）无原子保护 → ❌ |
| 4.3 | 是否有死锁风险？（多锁获取顺序不一致） | 不同代码路径以不同顺序获取锁 → ⚠️ |
| 4.4 | 计数器/序列号生成是否原子操作？ | 非原子自增 → ❌ |
| 4.5 | 缓存是否有并发失效/击穿风险？ | 热点 key 过期时大量回源 → ⚠️ |

## 五、安全

| # | 检查项 | 判定标准 |
|---|--------|---------|
| 5.1 | 用户输入是否经过校验和清理？ | 未校验直接使用 → ❌ |
| 5.2 | 是否有 XSS 风险？（前端直接渲染用户输入） | innerHTML/dangerouslySetInnerHTML → ❌ |
| 5.3 | 敏感数据是否加密存储？日志是否脱敏？ | 密码/Token 明文存储或打印 → ❌ |
| 5.4 | 认证和授权是否覆盖所有端点？ | 需认证的端点无 auth 中间件 → ❌ |
| 5.5 | 是否有 CSRF 风险？（状态变更操作无 token） | 状态变更无 CSRF 保护 → ⚠️ |
| 5.6 | 文件上传是否有类型/大小校验？ | 无校验 → ❌ |
| 5.7 | 依赖库是否有已知 CVE 漏洞？ | `npm audit` / `pip audit` 有 HIGH/CRITICAL → ⚠️ |

## 六、错误处理与韧性

| # | 检查项 | 判定标准 |
|---|--------|---------|
| 6.1 | 异常是否正确捕获？是否有空 catch？ | 空 catch 吞掉异常 → ❌ |
| 6.2 | 错误信息是否对用户友好？（不暴露内部细节） | 直接返回 stack trace 给前端 → ❌ |
| 6.3 | 是否有资源泄漏？（文件句柄、连接未关闭） | try-with-resources 缺失 → ❌ |
| 6.4 | 是否正确处理了 null/undefined/空数组？ | 未判空直接使用 → ❌ |
| 6.5 | 是否有适当的日志记录？日志级别是否合理？ | 关键路径无日志 → ⚠️；全部 info 级别 → ⚠️ |

## 七、代码质量

| # | 检查项 | 判定标准 |
|---|--------|---------|
| 7.1 | 命名是否清晰？是否与前后端一致？ | 同概念不同名称前后端不一致 → ⚠️ |
| 7.2 | 函数是否过长？（> 50 行建议拆分） | > 100 行 → ⚠️ |
| 7.3 | 是否有重复代码？是否可以抽取公共方法？ | 3 处以上相似代码 → ⚠️ |
| 7.4 | 是否有硬编码的魔法数字/字符串？ | 除 0/1/-1 外的无命名常量 → ⚠️ |
| 7.5 | 注释是否准确？是否有过时的注释？ | 注释与代码行为不一致 → ⚠️ |

## 八、前端专项

| # | 检查项 | 判定标准 |
|---|--------|---------|
| 8.1 | 是否有不必要的重渲染？（React.memo/useMemo 缺失） | 大列表无优化 → ⚠️ |
| 8.2 | 大列表是否使用虚拟滚动？ | > 1000 项无虚拟滚动 → ⚠️ |
| 8.3 | 是否覆盖了 4 种状态？（loading/empty/error/normal） | 缺少任一状态 → ⚠️ |
| 8.4 | 是否有内存泄漏？（useEffect 未清理、事件未解绑） | 未 return cleanup → ❌ |
| 8.5 | 图片/静态资源是否优化？（懒加载、压缩、WebP） | 大图未压缩直接加载 → ⚠️ |

## 使用规则

1. **提测前强制执行**：在 `/flnext-sdd-submit` 阶段，推送 feature 分支后、创建 MR 前，AI 必须对 `git diff develop...feature/{name}` 范围内的所有变更运行此清单
2. **必须包含在 MR 描述中**：自检报告作为 MR 描述的一部分，不能省略
3. **❌ 不通过的项目必须说明**：不能只标 ❌ 不解释，必须给出具体位置和修复建议
4. **⚠️ 需关注的项目需要人工判断**：AI 标注后，由代码审查者和 MR 审批者决定是否需要修复
5. **可以按项目类型裁剪**：前端项目可跳过数据库/并发检查，纯后端项目可跳过前端专项
