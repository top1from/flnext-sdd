# 对抗性评审指南

> Phase 3b 架构评审及代码评审必须遵守。零发现 = 审核不充分 → 拒绝通过。

## 核心立场

```
假设: 设计/实现存在缺陷，直到证据证明合理
强制: 零发现 → REWORK（重新审查）
核心问题: "缺少什么？" 而非仅 "有什么问题？"
```

## 八维度评审框架

| 维度 | 检查要点 |
|------|---------|
| 1. 正确性 | 逻辑是否满足 Spec？边界条件是否覆盖？ |
| 2. 安全性 | 注入、越权、敏感信息泄露？ |
| 3. 性能 | N+1、内存泄漏、阻塞调用？ |
| 4. 可维护性 | 命名、耦合、魔法数字？ |
| 5. 可测试性 | 可单元测试？依赖可 Mock？ |
| 6. 兼容性 | 是否破坏现有接口/数据格式？ |
| 7. 容错性 | 失败场景处理？降级策略？ |
| 8. 规范一致性 | 风格、命名、架构约束一致？ |

## 发现分级

| 级别 | 含义 | 处理 |
|------|------|------|
| BLOCKER | 阻塞发布/开发 | 必须修复后才能 PASS |
| WARNING | 显著风险 | 需修复或记录接受理由 |
| SUGGESTION | 改进建议 | 可选，但零发现仍不合格 |

## 执行规则

1. 每个维度至少输出一条发现（可为 SUGGESTION）
2. 某维度无任何发现 → 标注 `[待加强]` 并重新审查
3. 评审报告写入 `docs/sdd/{feature_name}/review-report.md`
