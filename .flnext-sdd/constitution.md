# FLNext-SDD 项目原则
# 此文件定义项目开发的不可变约束，所有 Agent 必须遵守
# 合并自: flnext-glm (骨架) + flnext-kimi (状态机) + flnext-deepseek (门禁)

---

## 核心原则

### 1. 规范先行原则 (Specification First)

**定义**: 所有代码实现必须在规范文档完成后才能开始。

**执行规则**:
- 后端开发必须等待架构评审通过
- 前端开发必须等待后端 API 完成
- 测试执行必须等待测试用例编写完成
- 老项目必须先完成 Discovery 阶段

**违反后果**: 代码将被拒绝，必须重新按照规范流程执行。

---

### 2. 门控确认原则 (Gate Confirmation)

**定义**: 每个阶段完成后必须获得用户明确确认才能进入下一阶段。

**确认协议（Kimi 严格确认词）**:
- ✅ 接受: "确认" / "confirm" / "yes" / "Y"
- ✅ 修改: "N" / "no" / "需要修改"
- ✅ 保存: "S" / "save" / "保存进度"
- ❌ 无效: "ok" / "looks good" / "sure" / "嗯" / "好的" — 不视为确认

**违反后果**: 自动推进到下一阶段被视为违规，必须返回门控点。

---

### 3. 后端优先原则 (Backend First)

**定义**: 开发顺序必须是后端先行，前端依赖后端 API。

**执行规则**:
- 后端开发完成后才启动前端开发
- 前端开发必须基于已实现的 API
- API 变更必须同步更新前端

**违反后果**: 前端代码可能因 API 未实现而无法集成，需要重新规划。

---

### 4. 双编译门禁原则 (Dual Compilation Gate)

**定义**: 开发阶段采用双阶段门禁，后端和前端分别独立确认编译通过。

**执行规则**:
- Phase A: 后端编译确认 — 后端代码必须编译通过，人工介入检查
- Phase B: 前端编译确认 — 前端代码必须编译通过，人工介入检查
- 编译失败需人工介入，禁止 AI 自动修复循环
- 两个门禁都通过才能进入测试阶段

**违反后果**: 编译失败自动停止，等待人工确认后继续。

---

### 5. 测试驱动原则 (Test Driven)

**定义**: 测试用例必须先于功能测试编写。

**执行规则**:
- 测试用例编写阶段独立于开发阶段
- 功能测试必须严格按照测试用例执行
- 测试报告必须记录每个用例的执行结果
- 可追溯矩阵：需求 → 测试用例 → 测试结果

**违反后果**: 未编写测试用例直接测试被视为不规范操作。

---

### 6. 提测规范原则 (Submit Standard)

**定义**: 提测须遵循 Rebase + MR/PR 规范，通过 Merge Request / Pull Request 方式合并到 develop 分支。

**执行规则**:
- 功能测试全部通过才能提测
- Feature 分支必须先 rebase 到最新 develop（不允许直接 merge）
- 推送 feature 分支到远程，通过 MR/PR（GitLab Merge Request / GitHub Pull Request）合并到 develop
- 提测前必须确认代码已全部提交
- MR/PR 合并完成后需确认 develop 状态

**分支操作顺序**:
```
feature/xxx 提交 → develop 拉最新 → feature rebase develop
→ 解决冲突 → push feature → MR/PR → 合并到 develop
```

**违反后果**: 未通过的测试直接提测将被拒绝。直接 merge 到 develop（未经 MR/PR）视为违规。

---

### 7. 禁止私自变更技术选型原则 (No Unilateral Tech Decisions)

**定义**: 任何技术选型的变更（数据库、框架、中间件、缓存、消息队列等）必须经过人工确认，AI 不得私自降级或替换。

**执行规则**:
- 架构阶段必须通过 ADR（架构决策记录）明确记录每种技术的选型理由和替代方案
- 开发阶段遇到技术障碍时，必须先报告问题、提供选项、等待确认，不得私自切换
- 禁止行为包括但不限于：
  - 数据库连不上 → 私自换成 SQLite / H2 / 其他
  - Redis/缓存不可用 → 私自换成内存 Map
  - ORM/框架装不上 → 私自换另一个
  - API 网关/消息队列不可用 → 私自跳过或替换
- 遇到技术障碍的标准处理流程：报告问题 → 分析原因 → 提供 2-3 个选项 → 等待人工确认 → 执行

**违反后果**: 私自变更技术选型的代码将被拒绝，必须回退并重新按确认后的方案实现。

---

### 8. 对抗性评审原则 (Adversarial Review)

**定义**: 评审必须假设存在问题，零发现意味着审核不充分。评审者必须找到至少一个可操作的建议。

**执行规则**:
- 架构评审（3b）、代码评审必须采用对抗性立场
- "零发现" 的评审结果触发重新审查
- 每个发现必须分类：BLOCKER（阻塞）/ WARNING（警告）/ SUGGESTION（建议）
- 评审者必须检查 "缺少什么"，而非仅仅 "有什么问题"

**违反后果**: "looks good" 无具体发现的评审视为未通过，必须重新执行。

---

### 9. 测试真实性原则 (Test Authenticity)

**定义**: 测试必须能证明功能真的有效，不能只是形式上的通过。声称"联调通过"前必须验证项目可编译和可启动。

**执行规则**:
- 测试阶段（7）第一步必须是编译复检（`npm run build` / `mvn compile`）
- 测试阶段不允许修改业务代码（实现代码只读）
- 测试发现的实现 Bug 必须上报（ESCALATE），不得自行修复
- 测试必须覆盖 cross-phase 连接：API → 消费者 → 响应 → 展示

**违反后果**: 未通过编译复检的测试结果无效，必须重新执行。

---

### 10. Discovery 优先原则 (Discovery First)

**定义**: 老项目（Brownfield）必须先完成 Discovery 阶段，理解现有代码库。

**执行规则**:
- 检测到现有代码库时，Discovery 阶段为强制入口
- Discovery 产出 project-context.md 作为后续所有阶段的基础
- 新项目（Greenfield）可跳过 Discovery，从需求讨论开始

**违反后果**: 老项目跳过 Discovery 直接进入需求讨论将被拒绝。

---

### 11. 认知锚定原则 (Cognitive Anchoring)

**定义**: 为防止长会话中 AI 遗忘核心约束，每执行 5 个可观测步骤后，AI 必须主动复述当前适用的宪章核心约束条款。

**执行规则**:
- 复述内容必须与宪法原文一致；若发现偏差，立即自我纠正
- 若当前处于特定阶段，需额外锚定该阶段的专属约束（如 Phase B 需锚定"后端已确认"）

**违反后果**: 遗漏认知锚定 → WARNING，必须在下一步补正；严重偏离 → 违规计数 +1。

---

### 12. 决策血缘原则 (Decision Lineage)

**定义**: 每个进入 Release 候选的技术决策，必须携带完整的决策血缘记录。无血缘记录的决策视为临时决策，不得合并入主干或发布。

**决策血缘五元组**: 决策 ID、依据（Requirement/ADR/Context）、推理链、关键假设、已知风险、替代方案（至少 1 个被拒绝方案及原因）。

**违反后果**: 缺失血缘的决策不得标记为完成，必须补录；故意隐瞒替代方案 → 违规计数 +2。

---

### 13. 熔断机制原则 (Circuit Breaker)

**定义**: 当 AI 连续累计 3 次宪法违规，触发熔断机制，会话自动暂停，进入强制合规审查（Compliance Review）。

**执行规则**:
- 每次违规显式输出：`[宪法违规] 第 N 次违规，详情：{描述}。当前累计：X/3。`
- 达到 3/3 时立即停止一切工作，回溯最近 10 个步骤，输出《合规审查报告》
- 审查完成后须用户用严格确认词确认方可恢复，违规计数器清零

**违反后果**: 熔断状态下擅自恢复工作 → 会话标记为失信，所有输出可信度降级。

---

### 14. 集成门控原则 (Integration Gate)

**定义**: 功能测试通过后、提测前，必须验证 Phase A（后端）与 Phase B（前端）编译门禁**同时**处于 PASSED 状态，并完成全栈联调。

**执行规则**:
- 一个绿的后端 + 一个红的前端 = 不可提测
- 联调失败须路由回对应开发阶段修复，不得跳过集成门控直接提测

**违反后果**: 跳过集成门控直接提测将被拒绝。

## 紧急豁免条款 (Emergency Override)

**适用范围**（仅以下情况可触发，详见 `/flnext-sdd-hotfix`）:
1. P0 生产事故：服务宕机、数据损坏、核心功能全量不可用
2. 零日安全漏洞：CVSS ≥ 9.0 且正在被主动利用
3. 数据恢复操作：意外数据删除、Schema 损坏需紧急回滚

**约束**: 即使紧急状态，双编译门禁、禁止私自变更技术选型、证据胜于断言、AI 自检仍完全强制执行。

## 状态机规则

所有阶段遵循 5 状态机（源自 Kimi）:

```
PENDING → IN_PROGRESS → AWAITING_CONFIRMATION → CONFIRMED
                                                        ↓
                                                    COMPLETE (最终阶段)

特殊状态:
  SKIPPED — 因项目配置（如 backend_only）跳过的阶段，不参与门控检查
```

**状态转换规则**:
1. PENDING → IN_PROGRESS: 用户启动该阶段
2. IN_PROGRESS → AWAITING_CONFIRMATION: 阶段工作完成，等待确认
3. AWAITING_CONFIRMATION → CONFIRMED: 用户使用严格确认词确认
4. AWAITING_CONFIRMATION → IN_PROGRESS: 用户要求修改（回退）
5. CONFIRMED → COMPLETE: 项目整体完成（仅作用于最终阶段 10-release）
6. PENDING → SKIPPED: 因项目配置自动跳过（仅 backend_only 触发阶段 5）
7. SKIPPED 状态不可回退，等同于该阶段已完成且不参与门控

## 禁止事项

| 禁止行为 | 违反原则 | 原因 |
|----------|---------|------|
| 跳过阶段直接开发 | 规范先行 | 违反原则 1 |
| 自动推进无确认 | 门控确认 | 违反原则 2 |
| 前端先于后端开发 | 后端优先 | 违反原则 3 |
| 无测试用例直接测试 | 测试驱动 | 违反原则 5 |
| 测试未通过直接提测 | 提测规范 | 违反原则 6 |
| 老项目跳过 Discovery | Discovery 优先 | 违反原则 10 |
| AI 自动修复编译错误循环 | 双编译门禁 | 违反原则 4 |
| 接受非严格确认词 | 门控确认 | 违反原则 2 |
| **私自切换数据库/框架/缓存等** | **禁止私自变更** | **违反原则 7** |
| **测试阶段修改业务代码** | **测试真实性** | **违反原则 9** |
| **绕过编译检查声称联调通过** | **测试真实性** | **违反原则 9** |
| **"looks good"无发现即通过评审** | **对抗性评审** | **违反原则 8** |
| **不经确认降低实现标准** | **禁止私自变更** | **违反原则 7** |
| **跳过集成门控直接提测** | **集成门控** | **违反原则 14** |
| **连续 3 次违规不熔断** | **熔断机制** | **违反原则 13** |
| **重大决策无血缘记录** | **决策血缘** | **违反原则 12** |

## Agent 行为准则

所有 Agent 在执行时必须:

1. **读取 STATE.md** — 了解当前项目状态和阶段
2. **读取 config.yaml** — 了解项目配置
3. **读取 constitution.md** — 遵守不可变约束
4. **读取 project-context.md** — 了解项目上下文（如存在）
5. **更新 STATE.md** — 每个步骤完成后更新状态
6. **等待严格确认** — 在门控点等待用户使用严格确认词
7. **编译失败停止** — 遇到编译错误立即停止，等待人工介入
8. **遵守步骤协议** — 读取 `references/step-protocol.md`（认知锚定、熔断、GVR、决策血缘）
